## 3. Описание проекта

Проект представляет собой учебный мини-конструктор сайта. За основу была взята публичная страница TravelLine Tech. Главная цель работы — сделать так, чтобы часть контента на публичной странице можно было менять через отдельную админ-панель без ручного редактирования HTML-файла и без перезапуска сервера.

Публичная страница открывается по адресу:

```text
http://localhost:3000
```

Админ-панель открывается по адресу:

```text
http://localhost:3000/admin.html
```

### Выбранный стек технологий

| Технология | Для чего используется | Почему выбрали |
|---|---|---|
| Node.js | запуск серверной части проекта | позволяет писать сервер на JavaScript, как и клиентскую часть |
| Express | веб-сервер, маршруты, API, раздача статических файлов | простой и понятный фреймворк для учебного проекта |
| EJS | серверный рендеринг публичной страницы | удобно подставлять данные из JSON прямо в HTML-шаблон |
| JavaScript в браузере | логика админ-панели | не требует сложного frontend-фреймворка |
| JSON | хранение данных сайта | простое файловое хранилище без настройки базы данных |
| Multer | загрузка изображений сотрудников | готовое решение для приёма файлов на Express-сервере |
| node:test | автоматические тесты | встроенный тестовый инструмент Node.js без лишних зависимостей |

Такой стек подходит для практики, потому что он не перегружает проект лишней инфраструктурой. При этом он показывает полный цикл работы веб-приложения: сервер, публичная страница, админка, API, хранение данных и тестирование.

## 4. Описание админки

Админка находится в файле `project/public/admin.html`. Она состоит из нескольких секций, каждая секция отвечает за свой блок публичной страницы.

В админке есть следующие разделы:

| Раздел админки | Возможности |
|---|---|
| Hero | изменить заголовок, подзаголовок и пункты статистики |
| Команда | добавить сотрудника, удалить сотрудника, изменить имя, должность и фото |
| Клиенты | добавить или удалить клиента, изменить название и логотип |
| Вакансии | добавить, удалить или изменить вакансию, указать формат работы и ссылку |
| Плюшки | добавить, удалить или изменить преимущество компании |

Каждая часть админки вынесена в отдельный JavaScript-файл. Это удобно для командной работы: разные участники могут работать над разными блоками и реже получать конфликты при объединении изменений.

Принцип работы админки:

```text
Пользователь открывает /admin.html
JavaScript делает GET /api/data
Сервер отдаёт данные из data/data.json
Админка строит формы на основе полученных данных
Пользователь меняет поля и нажимает «Сохранить»
JavaScript отправляет POST /api/data
Сервер перезаписывает data/data.json
Публичная страница при следующем открытии показывает новые данные
```

Данные сохраняются не в базе данных, а в JSON-файле `project/data/data.json`. Для учебного проекта это подходящее решение, потому что JSON легко читать, редактировать и тестировать. При этом структура данных похожа на то, как данные могли бы храниться в базе данных.

Для загрузки фото сотрудников используется отдельный API-роут:

```text
POST /api/upload/team-photo
```

Фотография сохраняется в папку `project/public/img/team/`, а в JSON записывается путь к файлу. Сервер принимает только изображения и ограничивает размер файла до 5 МБ.

## 5. Особенности кодовой базы

### Ключевые технические решения

1. Серверный рендеринг через EJS.

Публичная страница не собирает данные через JavaScript после загрузки. Вместо этого Express читает `data/data.json`, передаёт данные в `views/index.ejs`, а EJS формирует готовую HTML-страницу. Это делает схему проекта простой и понятной.

2. Один JSON-файл как источник данных.

Файл `project/data/data.json` хранит данные всех управляемых блоков. Админка получает этот файл через API, изменяет нужную часть и отправляет обновлённый объект обратно на сервер.

3. Разделение логики админки по блокам.

Для каждого управляемого блока создан отдельный JS-файл: `admin-hero.js`, `admin-team.js`, `admin-logo.js`, `admin-vacation.js`, `admin-plushki.js`. Это упрощает поддержку проекта и делает код более понятным.

4. Отдельный модуль для работы с данными.

Файл `src/storage.js` содержит функции `readData`, `writeData`, `addTeamMember`, `deleteTeamMember`, `addClient`, `deleteClient`, `addVacancy`, `deleteVacancy`, `addBenefit`, `deleteBenefit`, `updateHero`. Эти функции можно тестировать отдельно от браузера и сервера.

5. Загрузка файлов через Multer.

Для блока «Команда» реализована загрузка фотографий сотрудников. Пользователь выбирает файл в админке, файл отправляется на сервер, сохраняется в `public/img/team/`, а в ответ сервер возвращает путь для записи в JSON.

### Структура API

| Метод | Адрес | Что делает |
|---|---|---|
| `GET` | `/` | читает JSON и отдаёт публичную страницу, срендеренную через EJS |
| `GET` | `/api/data` | возвращает все данные сайта в формате JSON |
| `POST` | `/api/data` | принимает новый JSON и сохраняет его в `data/data.json` |
| `POST` | `/api/upload/team-photo` | принимает фото сотрудника и возвращает путь к сохранённому файлу |

### Как публичная страница получает данные

Схема работы публичной страницы:

```text
Браузер открывает http://localhost:3000
Express обрабатывает маршрут GET /
Сервер читает project/data/data.json
Сервер передаёт данные в project/views/index.ejs
EJS подставляет данные в HTML
Браузер получает готовую страницу
```

Примеры данных, которые подставляются на страницу:

| Данные из JSON | Где используются |
|---|---|
| `hero.title` | скрытый основной заголовок страницы |
| `hero.subtitle` | крупный заголовок первого экрана |
| `hero.stats` | список статистики в блоке преимуществ |
| `team.members` | карточки сотрудников |
| `clients.items` | логотипы клиентов |
| `vacancies.items` | карточки вакансий |
| `benefits.items` | список преимуществ и бонусов |

## 6. Тестирование

Для проекта написаны автоматические тесты в файле:

```text
project/tests/storage.test.js
```

Тесты запускаются командой:

```bash
npm test
```

Используется встроенный модуль Node.js `node:test` и модуль `node:assert`.

### Что проверяют тесты

| Проверяемая часть | Что именно проверяется |
|---|---|
| `nextId` | корректное создание нового id для пустого и непустого списка |
| Команда | добавление сотрудника, удаление сотрудника, обработка пустого имени, удаление несуществующего id |
| Клиенты | добавление клиента, запрет пустого названия, удаление клиента |
| Вакансии | добавление вакансии, запрет пустого названия, удаление вакансии |
| Плюшки | добавление преимущества, запрет пустого заголовка, удаление преимущества |
| Hero | обновление заголовка и подзаголовка, запрет пустого заголовка |
| Чтение и запись JSON | сохранение данных во временный файл и обратное чтение без потерь |

### Результат запуска тестов

Результат:


### Какие проблемы тесты помогли выявить или предотвратить

Тесты помогают заранее проверить типичные ошибки:

| Возможная проблема | Как тесты помогают |
|---|---|
| добавление элемента без обязательного названия | проверяют, что функция выбрасывает ошибку |
| лишние пробелы в полях | проверяют обрезку пробелов у имени сотрудника |
| неправильная генерация id | проверяют, что новый id больше максимального существующего |
| удаление несуществующего элемента | проверяют, что данные не ломаются и функция возвращает `false` |
| ошибка при чтении или записи JSON | проверяют полный цикл `writeData` → `readData` |

## 7. Трудности и решения

Во время работы над проектом возникали задачи, которые требовали не только написания HTML и CSS, но и понимания связи между клиентской частью, сервером и хранением данных.

| Трудность | Как решили |
|---|---|
| Нужно было сделать редактируемую страницу без полноценной базы данных | Использовали JSON-файл `data/data.json` как простое хранилище |
| Нужно было связать админку и публичную страницу | Сделали API `GET /api/data` и `POST /api/data`, через которое админка читает и сохраняет данные |
| Нужно было выводить актуальные данные на главной странице | Использовали EJS: сервер читает JSON и подставляет данные в шаблон перед отправкой HTML браузеру |
| Нужно было не смешивать код всех блоков в одном большом файле | Разделили админку на несколько JS-файлов по секциям |
| Нужно было загружать фотографии сотрудников | Добавили Multer, проверку типа файла и сохранение изображений в `public/img/team/` |
| Нужно было убедиться, что функции работы с данными не ломаются | Вынесли логику в `storage.js` и покрыли её автоматическими тестами |

### Где помог AI

AI помогал быстрее разбираться в структуре проекта, формулировать понятные комментарии, объяснять работу Express, EJS, JSON-хранилища и тестов. Также AI был полезен при разбиении задачи на шаги, при подготовке функций для тестирования и при оформлении документации.

### Где разбирались самостоятельно

Самостоятельно нужно было понимать предметную область проекта, проверять результат в браузере, запускать сервер, смотреть, как изменения из админки отражаются на публичной странице, исправлять конкретные несоответствия в интерфейсе и принимать решения о том, какие блоки должны быть управляемыми.

Главный вывод по работе с AI: он ускоряет разработку и помогает не застревать на рутинных местах, но всё равно нужно проверять код, понимать архитектуру и запускать проект самостоятельно.
